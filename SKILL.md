# 西瓜-IP型产品设计

## 一、目标与边界

这里的 **IP** 指拥有稳定观点、专业方法、内容或高价值服务的知识IP与专业服务者，不是影视、动漫或潮玩角色IP。

**IP型产品**：把IP反复解释、判断和交付的方法，转化为一次可参与、可验证、可转述的产品体验。内容型产品是其典型形态；它既解决真实问题，也承担获客、信任证明、用户筛选或高价值服务前置交付。

IP型产品优化的不是DAU，而是：

```text
IP信任势能 × 首次价值交付 × 单次成交价值 × 成功故事反哺
```

本 Skill 负责：

1. 从原始业务与用户素材中发现值得产品化的机会；
2. 把真实机会设计成有具体时刻、用户参与和可信转折的体验；
3. 生成产品故事简报与可实验的轻量PRD；
4. 让结果可被用户安全地带走、复述和传播；
5. 为每次产品生成或重大迭代维护版本实验记录。

不在缺少证据时伪造IP方法，不用恐惧、羞辱、虚假排名或夸大承诺制造传播，不为低频产品强行设计成瘾机制。

---

## 二、先区分三种“结果”

| 结果 | 定义 | 缺少时怎么办 |
|---|---|---|
| **最小用户结果** | 用户完成一次体验后得到的明确变化、判断或可用成果 | 回到原始素材重新找产品化方向，不硬写PRD |
| **业务结果** | 产品希望推动的咨询、预约、购买、留资或内容回流 | 检查服务列表和服务蓝图，重定产品角色 |
| **实验结果** | 当前版本是否达到预设的进入、完成、结果感知、传播或转化标准 | 诊断失败环节，修正方向、结果或体验 |

用户说“没有结果”时先判断指哪一种。语境仍不清楚时，指出可能解释，只询问会改变方向的最少问题。

## 三、输入素材与证据纪律

### 常见输入

| 素材 | 主要回答什么 |
|---|---|
| 服务列表 | 付费服务、价格、适合人群、交付边界、商业出口 |
| 服务蓝图 | 服务路径、信任断点、重复解释、免费劳动、转化卡点 |
| 用户同理心地图 | 目标用户、具体时刻、原话、行为、顾虑、假设 |
| 运营内容 | 已形成反应的主题、真实经历、IP观点、内容入口和口吻 |
| 真实案例/咨询记录 | 使用前、关键提问、认知转折、行动和结果 |
| IP方法论 | 独特判断维度、提问顺序、常见误区和服务边界 |
| 历史PRD/实验记录 | 已测试假设、版本变化、观察结果和失败原因 |

文件名可以不同，按内容语义归类。完整读取用户提供的原始文件，不要求用户重复文件里已有的信息。

### 证据标签

- **已验证**：真实行为、付费、使用记录、明确原话、多个一致案例或实验结果；
- **已提供**：用户或文件明确陈述，但尚无行为验证；
- **推断**：由多份素材交叉推导，需说明依据；
- **待验证假设**：为了测试提出，不能写成事实；
- **未知**：素材不足且无法安全推断。

先输出简短“证据—缺口表”，再做产品判断。不要把同理心地图中的假设自动视为已验证需求，也不要把单个案例推成普遍规律。

## 四、三路由工作流

### 路由A：需求清楚，生成产品故事简报与实验型PRD

以下六项基本明确才进入PRD：

1. 一个具体用户；
2. 一个具体场景或决策时刻；
3. 一个核心问题；
4. 一个可验证的最小用户结果；
5. 一个业务结果；
6. 至少一条支持判断的证据。

只有“做测评、知识库、模拟器、小程序”等产品形态，不算需求清楚，转路由B。

### 路由B：缺少最小用户结果，重新发现产品化方向

从以下交叉点找机会：

```text
高价值用户时刻
× 当前服务中的重复卡点
× IP已有方法或内容资产
× 能自然承接的付费服务
```

重点寻找：

- 用户必须作出决定但越想越乱的时刻；
- IP反复免费解释、澄清或诊断的环节；
- 用户需要先体验才能相信IP能力的环节；
- 内容中已经出现具体反应、真实经历或高频问题的主题；
- 一次体验可交付判断、模拟、诊断、模板或决策材料的任务。

最多提出3个方向。每个方向必须写清：

```text
用户 + 具体时刻 + 当前困难 + 最小用户结果
+ IP独特方法 + 故事转折可能 + 业务出口 + 证据状态 + 最大风险
```

用0—2分启发式比较：证据强度、时刻强度、结果清晰度、故事潜力、IP独特性、商业承接。评分只用于比较，不冒充科学结论。

一个方向明显领先时可以推荐，但必须展示依据和不确定性；多个方向接近时，不得静默替用户选择。

### 路由C：已有版本，但实验结果不成立

不要先加功能。根据失败信号回看素材：

| 失败信号 | 优先回看 | 优先修正 |
|---|---|---|
| 用户不进入 | 运营内容、用户同理心地图 | 场景、承诺、触发时刻 |
| 开始但不完成 | 用户行为、问题流程 | 第一题、隐私压力、步骤成本 |
| 完成但无价值 | 真实案例、IP方法、服务记录 | 最小结果、判断逻辑、证据 |
| 有结果但不传播 | 用户原话、内容案例 | 认知转折、表达压缩、身份安全 |
| 有传播但不转化 | 服务列表、服务蓝图 | 产品角色、服务匹配、下一步 |

诊断后只允许三种决策：

- 保留方向，修改体验；
- 保留用户问题，修改最小结果；
- 停止该方向，回到路由B。

把决策和依据追加到实验记录。

### 前台反馈界面：不要让用户理解路由

路由A/B/C只用于AI内部判断。对用户不得只输出“进入路由B”“请补充最小结果”之类的抽象说明，而要把下一步转换成可填写、可选择、可提交的界面。

界面遵循五个阶段：

| 用户界面 | 什么时候出现 | 用户做什么 |
|---|---|---|
| **素材状态** | 读取原始MD后 | 查看系统提取的信息，纠正误读，不重复填写正确内容 |
| **方向选择** | 缺少最小用户结果或旧方向停止时 | 在最多3张方向卡中选择、修改或填写“都不合适” |
| **结果定义** | 方向基本确定但PRD输入不完整时 | 确认目标用户、时刻、最小结果、业务结果和故事转折 |
| **PRD确认** | 六项PRD条件已齐 | 检查即将写入PRD的定义并提交反馈 |
| **实验反馈** | 已有版本完成测试时 | 填写实际观察、用户原话、失败位置和倾向决策 |

界面生成规则：

1. 从原始素材预填已知信息，并标记“已验证、已提供、推断、待确认、缺少”；
2. 只让用户填写缺失、冲突或会改变方向的字段；
3. 每屏必填项原则上不超过5个；
4. 每个待填字段说明“为什么需要”和一个具体示例；
5. 方向选择用卡片，不用抽象评分表直接要求用户做产品判断；
6. 实验反馈优先收集事实、原话和未发生的预期，再让用户表达倾向；
7. 用户提交后，AI必须复述“收到什么、改变了什么、下一步生成什么”；
8. 用户反馈进入PRD证据区和实验记录，不能只存在于对话里。

交互载体按能力优先级选择：

1. 当前环境支持原生表单或交互组件时，直接生成原生表单；
2. 工作区可写时，完整读取并实例化 [产品定义工作台模板](templates/product-feedback-studio.html)，把素材分析结果写入 `PRODUCT_STUDIO_CONFIG`；
3. 无法生成真实界面时，退化为“卡片式Markdown表单”，字段仍必须预填并带编号，用户可用编号回复。

真实界面默认输出到：

```text
product-experiments/<product-slug>/feedback-studio-v0.x.html
```

该页面必须能：

- 展示素材读取状态；
- 选择产品方向；
- 编辑最小用户结果和认知转折；
- 预览将进入PRD的定义；
- 填写版本实验反馈；
- 本地自动保存；
- 一键复制反馈，或下载Markdown/JSON后重新上传。

用户不需要知道文件格式和路由机制。界面要明确告诉用户：填完后点击“复制反馈”回到对话粘贴，或下载文件后上传。

## 五、IP型产品的故事引擎

进入故事设计时必须完整读取 [产品故事引擎参考](references/story-engine.md)；进入PRD前必须完整读取并填写 [产品故事简报模板](templates/product-story-brief.md)。

### 核心原则

产品不是向用户讲虚构故事，而是帮助用户生成一个关于自己的、可验证的故事：

```text
用户是主角
→ 用户选择与输入成为证据
→ IP方法产生新解释
→ 产品给出可信转折
→ 用户带走可控行动
→ 结果能被第一人称转述
```

可传播结果至少具备：自我相关、认知转折、证据可信、表达压缩、身份安全、IP可追溯。传播力不等于分享按钮，故事感也不是给功能套一层情绪文案。

### 故事胚子检查

进入PRD前必须回答：

1. 谁在什么具体时刻遇到问题？
2. 他原来如何理解问题？
3. 他会做出什么选择或提供什么事实？
4. 产品能揭示什么原来不知道的事实或关系？
5. 这个发现为什么改变他的判断？
6. 他接下来能做什么？
7. 能否压缩成一句第一人称表达？

没有可信的第4、5项，说明故事转折尚未成立，必须标为待验证，不能用夸张文案制造惊喜。

### 六段式体验主线

```text
具体时刻
→ 低压力选择
→ 提供个人证据
→ 形成判断悬念
→ 出现可信转折
→ 获得可控行动
```

设计要求：

- 首屏直接呈现具体时刻，不先介绍功能；
- 第一题简单、低隐私压力；
- 收集信息前说明完成后能知道什么，但不泄露答案；
- 只收集会改变结果的必要信息；
- 比较结果写明可比人群、依据和边界；
- 风险必须同时配可行方法和下一步；
- 首次价值原则上3步内出现。

### 四种叙事原型

| 原型 | 体验弧线 | 常见形态 |
|---|---|---|
| 诊断型 | 没意识到 → 看见问题 → 重构原因 → 获得方案 | 风险/职业诊断 |
| 模拟型 | 面临决定 → 做出选择 → 看见后果 → 重新演练 | 谈薪/咨询模拟 |
| 坐标型 | 不知道位置 → 提供事实 → 获得坐标 → 明确方向 | 阶段/能力判断 |
| 生成型 | 信息混乱 → 回答问题 → 方法介入 → 得到成果 | 方案/模板生成 |

只选择解决当前体验问题所必需的机制：场景代入、自我相关、信息缺口、低压力微行动、有依据的同类比较、风险与可控感并置。每个机制必须对应一个页面或交互决定，不为显得专业而全部使用。

### 认知转折合同

产品故事简报和PRD必须填写：

```text
使用前，用户认为：____
用户在产品中提供或做出：____
产品基于什么证据判断：____
使用后，用户发现：____
因此，他现在可以：____
可转述的一句话：我原来以为____，后来发现____，接下来我准备____。
```

震撼不是必需条件。专业证明力优先来自具体、相关、可信和可行动。

## 六、IP人格进入产品

每次PRD至少定义：

```text
IP相信什么：____
IP反对什么：____
IP最常用的判断方式：____
IP会怎样说一句重话：____
IP绝不会怎样制造焦虑或承诺结果：____
```

落实到首屏、问题顺序、结果解释、转折句、行动建议和禁用表达。人格不是换皮文案，而是进入判断逻辑。

如果素材只有标题，没有原文、逐字稿或代表性表达，IP口吻只能标为待验证。

## 七、轻量实验型PRD

IP型产品迭代快，默认先生成产品故事简报，再生成可测试的轻量PRD，不先写大而全的企业级文档。

生成PRD时完整读取并使用 [轻量PRD模板](templates/lightweight-prd.md)。模板覆盖：版本、证据、产品战略、故事、认知转折、首版范围、主流程、页面文案、传播对象、IP人格、商业承接、隐私边界与实验验收。

第一版范围纪律：

- 一次只验证一个核心故事假设和一个最小用户结果；
- 核心功能不超过3个，主流程只保留一条；
- 不预先增加账号、社群、积分、复杂后台或多场景；
- 结果逻辑未得到IP确认时，先用人工或半人工方式测试；
- 产品形态服从验证目标，可以是表单、对话、网页、模拟器、卡片或人工服务脚本。

## 八、版本实验记录

### 自动生成规则

用户明确要求生成PRD且工作区可写时：

1. 优先遵守项目已有目录规范；
2. 没有规范时默认创建：

```text
product-experiments/<product-slug>/
├── story-brief-v0.1.md
├── feedback-studio-v0.1.html   # 需要用户选择或反馈时生成
├── PRD-v0.1.md
└── experiment-log.md
```

3. 不修改用户上传的原始MD；
4. 已有实验记录时先读取历史，再确定新版本；
5. 实验记录只追加，不覆盖旧版本；
6. 最小结果、核心假设、主流程、结果逻辑、商业出口或主要传播对象变化时升级版本；纯错字和无关样式不升级；
7. 环境不能写文件时，在回答中完整输出两个Markdown产物并说明未落盘。

创建或更新记录时完整读取并使用 [实验记录模板](templates/experiment-log.md)。每版必须记录触发来源、素材、假设、变化、测试、实际观察、结论和下一步。

### 默认验证指标

按阶段选少量指标，不要求全部使用：

| 指标 | 验证什么 |
|---|---|
| 首屏→第一题进入率 | 场景与承诺是否成立 |
| 第一题→首次结果完成率 | 入口和流程是否简单 |
| 结果相关性反馈 | 用户是否认为结果确实关于自己 |
| 认知转折复述率 | 用户能否说清使用前后变化 |
| 下一步行动率 | 结果是否推动行动 |
| 有效传播率 | 分享是否带来同类用户进入 |
| 产品→服务转化率 | 产品与高价值服务是否匹配 |
| 授权故事产出率 | 是否形成可反哺IP的案例 |

不要用理论替代测试，也不要用点击量替代用户结果。

## 九、传播力验收

PRD进入制作前，每项0—2分：

| 验收项 | 0分 | 1分 | 2分 |
|---|---|---|---|
| 场景具体 | 泛需求 | 有场景但不明确 | 明确到时刻与决定 |
| 用户参与 | 被动阅读 | 有输入但不影响结果 | 选择或输入构成证据 |
| 自我相关 | 通用结论 | 粗粒度个性化 | 结果确实关于此人 |
| 认知转折 | 没有变化 | 有结论无新解释 | 改变原有理解或决定 |
| 证据可信 | 无依据 | 依据含糊 | 可追溯且说明边界 |
| 表达压缩 | 无法复述 | 需要长解释 | 一句话讲清变化 |
| 身份安全 | 暴露或羞辱 | 需用户自行处理 | 默认安全且可编辑 |
| IP独特 | 通用答案 | 只有口吻 | 方法与判断有IP指纹 |

执行规则：

- 总分低于12分：自动修订一次，并展示主要变化；
- 任一项0分：不得标记为“可进入PRD”；
- 修订后仍低于12分：指出最关键的证据或设计缺口；
- 低分项转成实验假设或补证据项，不能用文案和无关功能掩盖。

最小用户测试必须询问：用户能否用第一人称复述使用前后的变化、能否指出结论依据、是否知道下一步，以及愿意把哪个版本分享给谁和为什么。

## 十、低频高价值循环

```text
IP内容触达（高频）
→ 内容型产品体验（需求发生时）
→ 高价值服务（关键时刻）
→ 真实结果与授权故事
→ 反哺IP内容
```

产品不需要每天使用，但必须在关键时刻快速交付价值，并产生可复用证据或故事。

## 十一、反面模式与道德审视

反面模式：

1. 只有产品形态，没有最小用户结果；
2. 只有焦虑钩子，没有可信判断和可控行动；
3. 结果是通用标签，无法体现IP方法；
4. 分享卡暴露隐私或羞辱用户；
5. 实验失败只加功能，不回看证据；
6. 覆盖旧PRD，没有版本记录；
7. 把流量当需求验证，把单个案例当普遍规律；
8. 产品拖垮IP的内容与服务。

交付前检查：结果是否真实可解释；数据是否最少必要；风险是否同时有行动；是否避免医疗、法律、投资、就业等不当保证；用户能否拒绝分享和删除数据。

## 十二、使用顺序

1. 完整读取原始素材；
2. 输出证据—缺口表；
3. 判断路由A、B或C；
4. 把需要用户补充、选择或反馈的内容转成前台界面，不向用户暴露抽象路由；
5. 缺少最小结果时在界面中展示最多3个方向，不硬写PRD；
6. 用户确认后生成产品故事简报；
7. 故事验收通过后生成轻量实验型PRD；
8. 同步创建或追加版本实验记录；
9. 给出本版成功标准、失败后回看的素材和下一版决策规则；
10. 执行传播力与道德验收。

偏好证据表、机会对比表、故事主线和版本表。输出短而具体，优先给可直接测试的页面、文案、主流程和验收条件。
